# 电脑播放器项目总结

## 1. 项目背景

这个项目的目标，是做一个运行在 Windows PC 上、足够轻量、功能清晰、可直接交付使用的本地音频和视频播放器。

需求一开始并不复杂，核心是播放器最常用的交互能力：
- 进度条
- 快进快退 10 秒
- 播放 / 暂停
- 多个倍速
- 音量控制

在此基础上，又补充了首版真正必要的 MVP 功能，包括：
- 打开本地文件
- 拖拽文件播放
- 全屏
- 静音
- 当前时间 / 总时长显示
- 键盘快捷键
- 播放结束状态处理
- 错误提示
- 设置持久化

## 2. 项目理念

这个项目不仅是做一个播放器，也是一次“AI 协助编程”的完整实践。核心理念有三点：

- 先收敛目标，再开始编码
- 先做 MVP，再逐步完善体验
- 把 AI 当作协作开发者，而不是只回答问题的工具

在整个过程中，AI 主要承担这些角色：
- 帮助梳理需求和边界
- 帮助做技术选型
- 帮助拆模块和搭骨架
- 直接编写代码
- 协助定位 bug 和做回归修复
- 完成构建、发布和交付

## 3. 技术选型

最终采用的技术栈是：
- `C#`
- `WPF`
- `libmpv`
- `.NET 8`

选择理由如下：

### 3.1 为什么不是 Python

这个项目的目标是“轻量桌面播放器”。Python 在快速原型上很方便，但最终打包体积、桌面集成和播放器内核封装都不如 C# 更合适。

### 3.2 为什么是 C#

`C#` 在 Windows 桌面开发上效率高、学习成本适中、工程维护相对轻，适合快速做出可交付的桌面程序。

### 3.3 为什么是 WPF

`WPF` 能比较自然地完成桌面 UI、快捷键、拖拽、窗口控制、全屏切换等需求，也适合这类工具型软件。

### 3.4 为什么是 libmpv

播放器最难的部分不是按钮，而是底层解码和播放控制。`libmpv` 是成熟的播放内核，适合快速搭建可靠的本地播放器。

## 4. 开发过程

### 4.1 需求定义

项目先从用户直接提出的需求开始，然后逐步扩展成 MVP 清单。这个过程并不是一上来就写代码，而是先明确：
- 要做桌面版，不做浏览器版
- 要做 Windows 可执行程序
- 要先保证“能用”，再逐步优化交互

### 4.2 项目搭建

完成了基础工程初始化，包括：
- 创建解决方案和 WPF 项目
- 配置 `.NET 8`
- 引入 `libmpv` 相关依赖
- 配置 `x64` 目标平台
- 处理发布时需要的原生动态库复制

### 4.3 功能实现

逐步完成了首版播放器的核心功能：
- 打开文件
- 拖拽播放
- 播放 / 暂停
- 快进 / 快退
- 倍速切换
- 音量控制
- 静音
- 全屏
- 快捷键
- 设置持久化

### 4.4 测试与迭代

整个过程并不是一次性成功，而是典型的“实现 -> 测试 -> 出问题 -> 修复 -> 回归验证”的迭代过程。

这也是 AI 协作开发最有价值的部分：它不仅参与生成代码，也持续参与问题定位和收敛。

## 5. 关键问题与修复

项目中实际遇到过多轮问题，比较典型的有：

### 5.1 程序启动后闪退

原因集中在 `libmpv` 的加载顺序和动态库命名兼容问题上。修复后程序可以稳定启动。

### 5.2 文件无法打开

问题不在测试媒体本身，而在封装层的 helper 调用不稳定。最终改成更底层的 `mpv command` 调用，文件加载恢复正常。

### 5.3 快进、快退、快捷键不生效

这部分问题来自播放器控制命令与 UI 输入链路没有完全对齐，后续统一改为底层命令和更稳定的键盘事件处理。

### 5.4 中文输入法下快捷键失效

`M` 和 `F` 被输入法预编辑截走。最终通过识别 `ImeProcessedKey / SystemKey / Key` 的方式修复。

### 5.5 进度条点击跳转问题

这是整个项目中修得最久的一类问题。最初方案依赖 WPF 原生 `Slider` 的点击行为，但实际存在点击回弹、跳转不生效等问题。最终改为自定义 seek 条，直接由程序计算点击位置与目标时间，问题才真正稳定解决。

## 6. 项目成果

最终得到的是一个可以运行和发布的 Windows 本地播放器，具备完整的首版核心功能，并且已经能够：
- 打开本地视频和音频
- 正常播放与暂停
- 正常快进快退
- 正常进行进度跳转
- 在中英文输入法下使用快捷键
- 打包为可分发的发布目录

发布产物位于：
- `artifacts/publish/win-x64`

其中可执行文件为：
- `MyPlayer.App.exe`

## 7. AI 协助编程的经验总结

这次项目比较典型地体现了 AI 协作开发的优势：

- 在项目前期，适合做需求收敛和技术选型
- 在开发中期，适合做模块拆解和代码落地
- 在调试阶段，适合持续定位问题、提出假设并快速迭代修复
- 在交付阶段，适合补文档、整理发布说明和开源材料

同时也有一个很实际的经验：  
AI 不是“一次性写完全部代码就结束”，而是在不断验证、不断修正中逐步接近正确结果。尤其是桌面 UI、原生库集成、输入事件链路这类问题，通常需要多轮排查。
